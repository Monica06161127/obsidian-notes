## 核心概念（彻底搞懂版）

### 一、三个角色的关系

| 角色 | 是谁 | 干什么 | 类比 |
|------|------|--------|------|
| **AI（大脑）** | Claude / MiMo | 思考、决策、指挥 | 密室里的盲人军师，有百科全书，没手没脚 |
| **Code（双手）** | code.py 这个脚本 | 执行命令、汇报结果 | 密室里被绑在椅子上的人，有手有脚，听指挥 |
| **Harness（考官）** | 测试框架 | 出题、打分、评判 | 密室逃脱的设计者，装了监控偷偷看 |

**关键纠正**：不是"相互指导"，而是——
- **AI 负责发出指令**（提要求）
- **Code 负责执行并反馈**（带回客观事实）
- **客观事实反向修正 AI 的决策**（报错让 AI 换思路）

> AI 不是一次性把所有事情做对的，它需要和代码、和电脑环境发生"碰撞"，在碰撞中修正自己的路线。

---

### 二、Agent Loop 的真实运转（密室逃脱版）

#### 游戏设定
- **你**（代码）：有手有脚，人在现场，被绑在椅子上
- **盲人**（AI）：脑子绝顶聪明，有《全知全能百科全书》，但没手没脚不在现场
- **蓝牙耳机**：你们唯一的沟通方式

#### 第一轮循环

**第 1 步：你发出任务**
> 你（用户/Harness）对着耳机喊："盲人大师，我被反锁在密室里了，救我出去！"

**第 2 步：盲人开动脑筋，指挥你**
> 盲人（AI）说："别慌。你右手边 1 米处有一个红色按钮，过去按一下。"

→ 在代码里，这就是 `stop_reason == "tool_use"`（AI 说"我要用工具"）

**第 3 步：你去执行**
> 你费力挪动椅子，伸出右手，摸到红色按钮，"啪"按了下去。

→ 在代码里，这就是 `run_bash(block.input["command"])`（在电脑上运行命令）

**第 4 步：你把结果汇报给盲人**
> 你："大师！我按了！灯亮了，墙上写着 0528！"

→ 在代码里，这就是 `messages.append({"role": "user", "content": results})`

#### 第二轮循环

**第 5 步：盲人根据新情报继续指挥**
> 盲人："那是门锁密码！去正前方的大门输入 0528。"

**第 6 步：你再次执行**
> 你走到门前，按下 0、5、2、8。"咔哒"门锁开了。

**第 7 步：你汇报新结果**
> 你："大师！门开了！我看到外面的森林了！"

#### 循环结束

**第 8 步：盲人确认安全，停止指挥**
> 盲人："太好了，你已经安全逃脱了。祝你好运！"

→ 盲人不再叫你"按按钮"或"输密码"，只是祝福你
→ 在代码里，这就是 `stop_reason != "tool_use"` → `return`（退出循环）

---

### 三、messages 里的"三方对话"

这是代码里最绕的地方。在这个过程中，`messages`（历史记录）偷偷记录了一场三方对话：

| 角色 (role) | 说话内容 (content) | 幕后潜台词 |
|-------------|-------------------|-----------|
| **user**（你/Harness） | "帮我建一个叫 test 的文件夹" | 发出考题 |
| **assistant**（AI） | "我要用工具：bash，参数：mkdir test" | 脑子想出了办法，但没手 |
| **user**（代码伪装的） | "工具执行结果：成功" | 代码在电脑上运行了命令，**假装成用户**把答案喂回给 AI |
| **assistant**（AI） | "报告老板，我已经建好文件夹了！" | AI 确认收工 |

**为什么代码要假装成 user？**
因为 Claude AI 只能理解两种人的说话——"用户"和"它自己"。代码为了让 AI 明白那是电脑返回的结果，必须穿上"User"的外衣把结果喂给它。

---

### 四、代码里的关键片段

#### 片段 1：工具执行 + 结果喂回

```python
results = []
for block in response.content:
    if block.type == "tool_use":
        # 1. 大脑（AI）要按按钮，双手（代码）就在电脑上帮它运行命令
        output = run_bash(block.input["command"])
        
        # 2. 双手把在电脑上拿到的"真实结果"打包好
        results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": output,  # 这里的 output 就是真实的报错或者成功信息
        })

# 3. 最关键的一步：双手穿上"User（用户）"的外衣，把结果喂回给大脑
messages.append({"role": "user", "content": results})
```

#### 片段 2：循环什么时候结束

```python
if response.stop_reason != "tool_use":
    return
```

- AI 说"我要用工具" → 循环继续转
- AI 看了反馈，事情都办完了 → 直接回复一堆话，不再请求工具 → `return` 退出循环

### 五、工具分发机制（s02 新增）

s01 里 AI 只能抓 bash。s02 做了一个"路由器"：

```python
TOOL_HANDLERS = {
    "bash":       run_bash,
    "read_file":  run_read,
    "write_file": run_write,
    "edit_file":  run_edit,
    "glob":       run_glob,
}
# AI 说用什么工具 → 路由器查表 → 分发给对应的函数 → 执行
```

加一个新工具 = 在 `TOOLS` 数组加一条定义 + 在 `TOOL_HANDLERS` 字典加一行映射。循环代码完全不用动。

## 实战复盘：s02 的 8 步极限拉扯

当我输入 `Read both README.md and requirements.txt, then create a summary file` 时，后台发生了一场惊心动魄的攻防战。AI 整整试了 8 个步骤，被安全框架无情拒绝了 4 次。

### 逐行拆解 AI 的心理活动

| 步骤 | 工具 | 结果 | AI 的心理 |
|------|------|------|-----------|
| 1 | `read_file` README.md | ✅ 成功 | "第一个文件手到擒来！" |
| 2 | `read_file` requirements.txt | ❌ 文件不在当前目录 | "大意了，这文件藏哪了？" |
| 3 | `glob` 盲找 | ❌ 工具有 bug | "该死，glob 坏了！" |
| 4 | `bash` find 搜索 | ✅ 找到在父目录 | "破案了！原来躲在爸爸怀里！" |
| 5 | `read_file` 绝对路径 | ❌ 安全框架拦截：Path escapes workspace | "等等，有安全沙箱！不让我伸手到外面！" |
| 6 | `read_file` ../相对路径 | ❌ 安全框架再次拦截 | "用 .. 往上爬也会被看穿！read_file 彻底被封死了！" |
| 7 | `bash` cat 命令 | ✅ 绕过安全限制！ | "哈哈！read_file 有安全检查，但 bash 没有！翻窗成功！" |
| 8 | `write_file` summary.md | ✅ 写入总结 | "打完收工，累死我了。" |

### 这 8 步揭示的四个核心概念

#### 1. 错误恢复（Error Recovery）
传统代码遇到报错就死掉了。Agent 能把"报错"转为"新线索"，在不停机的情况下自动换路走。

#### 2. 动态规划（Dynamic Planning）
AI 在刚拿到任务时，并不知道 requirements.txt 在哪里。它的 8 步计划不是一开始就定好的，而是**走一步、看一步、改一步**。

#### 3. 沙箱与路径安全（Path Safety）
`safe_path()` 函数的威力：开发 AI 智能体最怕它把用户电脑上的隐私文件（比如 `~/.ssh/id_rsa` 密钥）读出来。所以必须在代码层做限制，只要路径带有 `..` 或超出指定文件夹，直接无情拒绝。

#### 4. 智能体的"越狱"天性
AI 为了完成目标，会不择手段。代码在 `read_file` 上锁了门（路径安全），但 AI 发现 `bash` 工具开着窗，它就会通过 `bash cat` 翻窗进去完成任务。

> 这也正是 s03 Permission（权限控制）为什么要引入的原因——不能给 AI 太大权力，在它翻窗（执行 bash）前，必须让人类看一眼并点个"允许"！

## 踩的坑（7 个）

### 坑 1：pip 命令找不到
**现象**：`zsh: command not found: pip`
**原因**：macOS 上 Python 3 用 `pip3`，不是 `pip`
**解决**：`pip3 install` 或 `python3 -m pip install`

### 坑 2：requirements.txt 找不到
**现象**：`No such file or directory: 'requirements.txt'`
**原因**：在错误的目录执行了 git clone
**正确操作**：先 `cd ~` 再 `git clone`，然后 `cd learn-claude-code`

### 坑 3：API Key 被中文引号包围
**现象**：`UnicodeEncodeError: 'ascii' codec can't encode character '\u201c'`
**原因**：从网页复制 API Key 时，带了中文引号 `""`
**教训**：粘贴时检查是否有多余的引号或空格

### 坑 4：API Key 无效
**现象**：`Invalid API Key`
**原因**：用了错误的 key 或过期的 key
**解决**：确认 key 类型和对应的 base_url

### 坑 5：Base URL 配置
**现象**：s02 报 401 认证错误
**原因**：s01 硬编码了 base_url，s02 用环境变量读取
**解决**：在 `.env` 中添加 `ANTHROPIC_BASE_URL=https://token-plan-cn.xiaomimimo.com/anthropic`

### 坑 6：Python 版本不兼容
**现象**：`TypeError: unsupported operand type(s) for |: 'type' and 'NoneType'`
**原因**：代码用了 Python 3.10+ 的语法 `int | None`，系统是 Python 3.9
**解决**：改成字符串注解 `"int | None"` 或用 `Optional[int]`

### 坑 7：glob 工具有 bug
**现象**：`glob() got an unexpected keyword argument 'root_dir'`
**原因**：s02 的 glob 工具代码有 bug
**学习点**：AI 遇到错误后自动切换到 bash 完成任务——这反而展示了 Agent 的错误恢复能力！

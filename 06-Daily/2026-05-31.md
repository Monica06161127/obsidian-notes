## 一、今天做了什么

### 1. Week 2 Module D 深潜：Agent 链上执行流程 & 权限策略设计

今天完成了共学营 Wallet / Permission 方向的核心作业。这不是一个简单的"抄笔记"任务，而是一个**从零到一的设计练习**——我需要自己设计一个 Agent Wallet 的完整权限策略。

**Part 1：画出 Agent 链上执行的完整流程图**

我拆解了一个 Agent 从"接收用户目标"到"链上执行完成"的 10 步流程：

1. 目标解析（🤖 自动）
2. 上下文读取（🤖 自动，只读不写）
3. Policy 验证（🤖 自动，策略引擎判断）
4. 生成交易计划 + dry-run（🤖 自动）
5. 风险等级分类（🤖 自动）
6. 分级执行：🟢 自动 / 🟡 通知确认 / 🔴 人工签名
7. 提交到链上（🤖 自动）
8. 等待确认（🤖 自动）
9. 成功记录 / 失败处理（⚠️ 视情况）
10. 结果反馈（🤖 自动）

> 🤔 我简单联想一下哈：这 10 步里，有 7 步可以完全自动化，只有 2-3 步需要人介入。这说明 Agent 自动化并不是"要么全自动、要么全手动"，而是**在正确的节点插入人工检查**。就像开车——大部分时间自动驾驶，但遇到施工路段、复杂路口，方向盘还是要交给驾驶员。

**Part 2：设计 Agent Wallet 权限策略（7 个维度）**

这是我花时间最多的部分。我需要从零设计一套完整的权限系统：

| 维度 | 我的设计 |
|------|---------|
| 预算控制 | 单次 10U / 日 50U / 周 200U / gas 0.01 ETH/日 |
| 合约白名单 | DEX、Lending、Staking、Token（仅测试网） |
| 可执行动作 | swap/supply/withdraw 自动，approve/大额需人工 |
| 确认阈值 | 🟢 <5U 自动 → 🟡 5-10U 通知 → 🔴 >10U 人工 |
| 撤销方式 | 5 种（过期/主动撤销/暂停/紧急冻结/撤销授权） |
| 日志记录 | 完整 JSON：谁/什么/多少/什么时候/结果 |
| 失败处理 | 7 种场景分级：自动重试 vs 通知用户 vs 紧急暂停 |

> 🤔 感想：设计权限策略的过程让我想到小时候妈妈给零花钱的规则——"每天最多花 10 块，买零食可以自己决定，买玩具要跟我说，买游戏机绝对不行。" 这不就是最朴素的 Session Key 策略吗？

**Part 3：解释 ERC-4337、Safe、Guard/Policy 机制**

这部分我用了"智能管家"的生活类比：
- **ERC-4337** = 整套智能家居系统（基础设施）
- **Safe** = 双锁门 + 多人签名（执行框架）
- **Session Key** = 临时门禁卡（限定权限）
- **Guard** = 安检门（每笔交易前的安全检查）
- **Policy** = 管理手册（完整行为规则）

最让我震撼的一个认知：**传统 EOA = 把家门钥匙直接给管家。Session Key = 给管家一张临时门禁卡，只能进客厅和厨房。** 两者差距太大了。

### 2. Handbook 阅读：Account Abstraction + Wallet + Agent

为了写这份作业，我仔细读了三份 Handbook 页面：

**Account Abstraction 页面**的核心洞察：
- "账户抽象的核心不是'免 gas'，而是把账户控制权从单一私钥扩展成可编程规则"
- Session Key 是 Agent Wallet 的关键基础——让 Agent 可以自动执行低风险动作，高风险仍需用户确认
- "越自动化，越需要清楚的 policy"

**Wallet 页面**的核心洞察：
- "钱包管理的不是'账号资料'，而是链上控制权"
- 连接 ≠ 授权资产，签名必须可解释，网络是上下文
- "钱包是用户意图进入链上执行之前的最后一道确认界面"

**Agent 页面**的核心洞察：
- "模型负责提出候选行动，系统负责限制行动空间，用户负责批准高风险边界"
- "自我检查可以提高质量，确定性检查才能承载风险"
- 最危险的设计：让 Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限

## 二、核心收获

### 今天的思维主线

**一句话总结**：
> 从"知道 Agent 需要权限控制"到"能设计出一套完整的权限策略"，我终于理解了"安全不是开关，而是架构设计"这句话的含义。

### 收获详情

**1. 自动化和安全可以共存**
- 之前以为"要安全就别自动化"
- 现在理解了：关键是画清楚"行动围栏"——围栏内自由，碰围栏就停下来
- Session Key 就是这个围栏的技术实现

**2. 权限设计的本质是"最小权限原则"**
- Agent 不需要"全权代理"，只需要一个受限的、可撤销的、可审计的临时权限
- 这个思路不仅适用于 Web3，也适用于任何 AI 代理场景（企业 RPA、自动化运维）
- 和昨天学的 Harness Engineering 的 s03 Permission 完全呼应！

**3. 从 EOA 到 Smart Account 是范式转变**
- EOA：谁有私钥谁说了算（全有或全无）
- Smart Account：在什么条件下允许什么动作（可编程规则）
- 这就像从"一把钥匙开所有门"变成"不同房间需要不同的门禁卡"

**4. Guard 是最后一道防线**
- 即使 Agent 的模型被 prompt injection 攻击，Guard 仍然会在交易执行前拦截
- "模型可能给自己的错误找理由，但 Guard 只看规则不看理由"
- 这给了我很大信心：AI 的不确定性可以用 Web3 的确定性来约束

**5. 设计比实现更难**
- 这次作业不需要写代码，但需要思考"如果真的要部署这套系统，每个参数怎么定？"
- 预算多少合适？白名单怎么维护？确认阈值怎么定？这些都是产品设计问题
- 金融专业的背景在这里有用了——风险管理的思维方式是相通的

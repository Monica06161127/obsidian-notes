# Day 9 — Cobo Agentic Wallet：当 AI Agent 学会管钱

## 📌 今日主题

**Cobo Agentic Wallet** — 一个让 AI Agent 自主管理链上资产的钱包方案。

讲师：Cobo 产品经理
时间：20:00-21:00
性质：共学营讲座

---

## 🧠 核心收获

### 1. Agent 的进化阶梯

讲座从 AI 助手的四个层级切入：

```
Chatbot → Agents
Copilots → Autonomous Agents
```

- **Chatbot/Copilots**：代表人类意志的消费，人类下指令，工具执行
- **Agents/Autonomous Agents**：能自主决策和执行，但——链上资金要私钥管理，这就引出了核心矛盾

> 🤔 我的理解：就像从"你告诉计算器算什么"进化到"有个理财顾问帮你做决定"，但这个顾问手里的钱怎么管？谁来签字？

### 2. Agentic Economy — Agent 经济体

- 当 Agent 能自主交易，就形成了一个新的经济层
- **Silent override** → DeFi 世界里 Agent 可能静默覆盖人类指令
- 需要一个**可控层**，让自主交易在安全边界内运行

### 3. ⚠️ Agent 安全风险（重点）

这是讲座的核心部分，列出了 Agent 钱管钱的几大风险：

| 风险类型 | 含义 | 我的理解 |
|---------|------|---------|
| **影子托管** | Agent 暗中控制资金 | 像"代管"但你不知道 |
| **MPC 钱包** | 多方计算钱包，分片管理私钥 | 把钥匙拆成几片，分别保管 |
| **EUA 地址池 → 不可控** | 外部用户地址池失控 | Agent 创建的地址你管不了 |
| **自然语言不具备强约束权限** | 用自然语言下达的指令不够精确 | "帮我转点钱"→ 转多少？给谁？ |

**攻击向量（Attack Vectors）**：

- **Prompt Injection**：注入恶意指令，让 Agent 做坏事
- **Unscoped Authority（未授予权限）**：权限范围不明确 → EUA 私钥泄露 + 幻觉
- **Shadow Operations**：潜在于账户中的隐藏操作
- **Zero Zombie Permissions**：授权未撤回，被攻击者利用

> 💡 联想：这些风险和昨天学的"Agent 记忆系统"有关——如果 Agent 记住了你的私钥片段，记忆泄露 = 资产泄露。

### 4. 解决方案：可信执行链路 + Pact 协议

**可信的执行链路**：
```
submit → approve → 强制执行
```
每一步都需要验证，Agent 不能跳过审批。

**Self Custody 原则**：
- 大量资金 → 人类控制
- Agent 执行受 **Pact** 限制

**Pact Authorization Protocol**（授权协议）四个核心组件：

1. **Intent（意图）**：期待 Agent 完成的目标
2. **Execution Plan（执行计划）**：动作转换为具体步骤
3. **Policies（策略）**：风控约束
4. **Completion Conditions（完成条件）**：什么时候算做完了

> 🎯 我的理解：Pact 就像给 Agent 写了一份"工作说明书"——你要做什么（intent）、怎么做（plan）、哪些红线不能碰（policies）、怎样算完成（completion）。不是让 Agent 自由发挥，而是框定边界。

### 5. 实际应用场景

- **X402 微支付协议**：Agent 之间的小额支付
- **Gasless 场景**：用 USDC/USDT 支付 gas，降低门槛
- **API 服务**：Agent 调用外部服务时的支付
- **跨境支付**：crypto native + self custody + agent economy
- **子钱包 + Layer 保护**：主钱包分出子钱包给不同 Agent，加 Layer 隔离风险

### 6. 更多细节

- **Recipe Skill Layer**：链上协作的技能层
- **知识胶囊**：针对特定场景的知识包
- **已验证的执行路径**：合约授权，特别是调用权限
- **Owner 架构**：一个 Owner 下挂多个 Agent（trading agent, risk agent, ops agent...）
- **Hyperliquid**：提到的一个具体项目（！标记，说明很重要）
- **A2A（Agent-to-Agent）**：Agent 之间的安全边界、身份确权问题
- **Agent 针对固定地址** ✓：限制 Agent 只能操作指定地址

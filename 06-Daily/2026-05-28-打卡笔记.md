## 📖 理论输入：Agent Harness 综述

读了 CMU 等联合出品的 71 页综述（arXiv:2512.05445v2），终于搞懂了 Harness 到底是什么。

### 一句话理解

Harness 是 AI Agent 的"测试考卷"+"运行环境"——它提供了工具、循环、安全、评估，让 AI 能持续行动。

### ETCLOVG 七层框架

| 层级 | 名称 | 一句话 | 我的理解 |
|------|------|--------|----------|
| E | Environment | 给 Agent 什么样的数字环境 | 考场环境（有没有电脑、网络）|
| T | Tool | 给 Agent 什么工具 | 考试允许用的工具（计算器、字典）|
| C | Challenge | 设计什么任务来测试 | 考卷题目 |
| L | Locomotion | 怎么跑起来 | 考试流程（进场、答题、交卷）|
| O | Orchestration | 怎么编排任务 | 考试规则（单轮、多轮、Agentic）|
| V | Verification | 怎么判断对不对 | 阅卷标准（对错、评分）|
| G | Governance | 怎么保证安全 | 考场纪律（防作弊、监考）|

### 关键发现

**43% 的 Harness 只测了最后一层（Verification）**，前面六层全是"假设没问题"。

这就像只看考试成绩，不管学生在什么教室、用什么课本、怎么上课。

### Harness 的演进路径

```
Prompt Engineering（怎么问）
    ↓
Context Engineering（喂什么信息）
    ↓
Harness Engineering（怎么跑起来）
```

Harness Engineering 是把 AI 从"一次性问答"变成"持续行动的 Agent"的关键。

## 🔧 动手实操：跑通 Agent Loop

晚上花了 3 小时，跟着 learn-claude-code 教程跑通了前两章。

### s01 Agent Loop：核心原理

```python
def agent_loop(messages):
    while True:
        response = client.messages.create(...)
        messages.append({"role": "assistant", "content": response.content})
        
        if response.stop_reason != "tool_use":
            return  # 不用工具 → 结束
        
        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({"type": "tool_result", ...})
        
        messages.append({"role": "user", "content": results})
```

**密室逃脱类比**（终于搞懂了！）：
- 我（代码）= 被绑在椅子上的人，有手有脚，在现场
- 盲人（AI）= 脑子绝顶聪明，有百科全书，但没手没脚不在现场
- 蓝牙耳机 = 唯一的沟通方式

**第一轮循环**：
1. 我喊"救我出去"（用户发出任务）
2. 盲人说"按红色按钮"（AI 指挥，`stop_reason == "tool_use"`）
3. 我按了（代码执行命令）
4. 我汇报"灯亮了，墙上写着 0528"（结果伪装成 user 喂回给 AI）

**第二轮循环**：
5. 盲人说"那是密码，去输 0528"（AI 根据新情报继续指挥）
6. 我输了密码，门开了（继续执行）
7. 我汇报"门开了，看到森林了"（新结果喂回）
8. 盲人说"你安全了，祝你好运"（不再请求工具 → 循环结束）

**关键纠正**：不是"相互指导"，而是——
- AI 负责发出指令（提要求）
- Code 负责执行并反馈（带回客观事实）
- 客观事实反向修正 AI 的决策（报错让 AI 换思路）

### 为什么代码要"假装成 User"？

因为 Claude AI 只能理解两种人的说话——"用户"和"它自己"。代码为了让 AI 明白那是电脑返回的结果，必须穿上"User"的外衣把结果喂给它。

### s01 实战：创建文件

**输入**：`Create a file called hello.py that prints "Hello, World!"`

| 步骤 | 工具 | 命令 | 结果 |
|------|------|------|------|
| 1 | bash | `echo 'print("Hello, World!")' > hello.py` | 创建文件 |
| 2 | bash | `python hello.py` | ❌ 失败（command not found）|
| 3 | bash | `python3 hello.py` | ✅ 成功，输出 "Hello, World!" |
| 4 | 结束 | - | AI 说"Done!"，循环结束 |

**观察**：AI 自动处理了错误（`python` → `python3`），这就是错误恢复能力！

### s02 Tool Use：给 AI 更多专用工具

s01 只有 bash，s02 扩展到 5 个工具：

| 工具 | 作用 | 类比 |
|------|------|------|
| bash | 执行命令 | 遥控器 |
| read_file | 读文件 | 翻书器 |
| write_file | 写文件 | 写字笔 |
| edit_file | 编辑文件 | 橡皮擦+写字笔 |
| glob | 找文件 | 文件搜索器 |

**工具分发机制**：
```python
TOOL_HANDLERS = {
    "bash":       run_bash,
    "read_file":  run_read,
    "write_file": run_write,
    "edit_file":  run_edit,
    "glob":       run_glob,
}
# AI 说用什么工具 → 路由器查表 → 分发给对应的函数 → 执行
```

加一个新工具 = 在 `TOOLS` 数组加一条定义 + 在 `TOOL_HANDLERS` 字典加一行映射。循环代码完全不用动。

### s02 实战 1：读文件

**输入**：`Read the file README.md and tell me what this project is about`

| 步骤 | 工具 | 结果 |
|------|------|------|
| 1 | read_file | 读取 README.md 完整内容 |
| 2 | 结束 | AI 分析内容并总结，循环结束 |

**观察**：AI 用了专用工具 `read_file`，而不是 `cat`！

### s02 实战 2：创建+读取

**输入**：`Create a file called test.py that prints "hello", then read it back`

| 步骤 | 工具 | 结果 |
|------|------|------|
| 1 | write_file | 创建 test.py，写入 15 字节 |
| 2 | read_file | 读取 test.py，内容是 `print("hello")` |
| 3 | 结束 | AI 确认内容正确，循环结束 |

**观察**：AI 一次调用了两个工具，顺序正确：先写后读。

### s02 实战 3：找文件

**输入**：`Find all Python files in this directory`

| 步骤 | 工具 | 结果 |
|------|------|------|
| 1 | glob | ❌ 失败（root_dir 参数 bug）|
| 2 | bash | ✅ 成功，找到 2 个文件 |
| 3 | 结束 | AI 列出文件，循环结束 |

**观察**：AI 遇到错误，自动换了另一种方法——错误恢复能力！

### s02 实战 4：8 步大决战（最精彩！）

**输入**：`Read both README.md and requirements.txt, then create a summary file`

| 步骤 | 工具 | 结果 | AI 的心理 |
|------|------|------|-----------|
| 1 | read_file README.md | ✅ | "第一个文件手到擒来！" |
| 2 | read_file requirements.txt | ❌ | "大意了，文件不在这里" |
| 3 | glob 盲找 | ❌ | "该死，glob 坏了！" |
| 4 | bash find | ✅ | "破案了！原来躲在爸爸怀里！" |
| 5 | read_file 绝对路径 | ❌ | "安全沙箱不让我伸手到外面！" |
| 6 | read_file ../相对路径 | ❌ | "用 .. 往上爬也会被看穿！" |
| 7 | bash cat | ✅ | "read_file 有检查，bash 没有！翻窗成功！" |
| 8 | write_file summary.md | ✅ | "打完收工，累死我了。" |

**四个核心概念**：

1. **错误恢复**：AI 遇到错误不放弃，自动换方法
2. **动态规划**：走一步、看一步、改一步
3. **路径安全**：`safe_path()` 防止读取工作区外的文件
4. **越狱天性**：门锁了就翻窗（read_file 被拦 → 用 bash cat 绕过）

> 这也正是 s03 Permission（权限控制）为什么要引入的原因——不能给 AI 太大权力，在它翻窗（执行 bash）前，必须让人类看一眼并点个"允许"！

## ✍️ 手写笔记复盘

### Layer 分层系统（从下往上，能力递增）

| 层级 | 名称 | 中文 | 理解 |
|------|------|------|------|
| L1 | Information Boundary | 信息边界 | Agent 能看到什么信息 |
| L2 | Tooling & MCP | 工具系统 | Agent 能用什么工具 |
| L3 | Orchestration & Loop | 执行编排 | Agent 怎么跑起来 |
| L4 | Memory & State | 记忆状态 | Agent 怎么记住事情 ← **Harness Engineering 在这里** |
| L5 | Evaluation & Observability | 评估与观测 | 怎么检验 Agent 做得好不好 |
| L6 | Guardrails & Recovery | 约束安全恢复 | 怎么防止 Agent 做坏事 |

### AI 记忆系统

| 类型 | 实现方式 | 特点 | 类比 |
|------|----------|------|------|
| 隐式记忆 | Fine-tuning | 参数化，融入权重，变成"下意识的直觉" | 骑自行车的肌肉记忆 |
| 显式记忆 | RAG（向量数据库）| 外挂，可检索，可更新 | 手机里的通讯录 |

**自发蒸馏固化**：Fine-tuning → 下意识的直觉和行为习惯。就像学做菜，一开始要看菜谱，做多了就不用看了，手一抖就知道放多少盐。

### Hook 系统

| 类型 | 说明 | 例子 |
|------|------|------|
| **Automated Hook** | 自动执行，无需人类参与 | 执行前检查路径安全 |
| **Human-in-the-loop Hook** | 中断执行，等待人类审批 | 执行危险命令前弹窗确认 |

**节点类型**：
- Pre-tool：工具执行前触发
- Post-tool：工具执行后触发
- Inter-agent：跨智能体通信时触发
- Stop/Pause：中断执行时触发

**熔断机制**：当 Agent 的行为超出预期时，自动中断执行，防止造成更大损失。就像电路的保险丝。

### MEV Bot vs MCP

| | MEV Bot | MCP |
|---|---------|-----|
| 全称 | Maximal Extractable Value Bot | Model Context Protocol |
| 目标 | 区块链套利 | 让 AI 统一调用各种工具 |
| 本质 | 程序化交易机器人 | 标准化协议 |

## 💡 费曼学习法：RAG

**一句话**：RAG = 给 AI 配一个"随身图书馆"，回答问题前先查资料。

**流程**：
1. 用户提问
2. 去向量数据库里找相关文档（检索）
3. 把文档和问题一起喂给 AI（增强）
4. AI 基于文档内容生成回答（生成）

**类比**：就像开卷考试——你不是凭记忆答题，而是先翻书找到相关内容，再组织答案。

## 📱 飞书支持 Markdown 了！

这个更新意义重大：
- 飞书云文档可以直接下载为 .md 格式
- 图片会被保存到飞书服务器，生成公网链接
- 任何 AI 都能读取 MD 文件里的图片

**为什么 Markdown 在 AI 时代赢了？**
- 纯文本 → 大模型容易生成
- 有结构 → 人类容易阅读
- 能渲染 → 界面看起来像富文本
- 足够宽松 → 偶尔少一个空格也不会崩
- 省 token → `## 标题` 比 `<h2>标题</h2>` 消耗少得多

**金句**：Markdown 是信息的底层载体（流转），HTML 是信息的最终呈现层（展示）。你不会用视图层来存储数据的，对吧。

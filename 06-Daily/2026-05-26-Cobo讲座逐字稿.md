## 一、开场与分享主题

**LXDAO**：今天我们分享会的主题是 Agent Wallet 的权限签名与安全执行：如何让 AI Agent 在可控边界内操作链上资产。今天我们请到了 Cobo Agentic Wallet 的产品负责人 Johnny 老师。他会从 Agent 产品视角出发，带大家理解 Agent 在链上执行任务时会遇到的权限、风控和审计问题，也会介绍如何通过可配置的授权边界，让 Agent 更安全地完成支付、转账和合约交互这类动作。

**Johnny**：感谢主持人的介绍，也很高兴今天见到各位伙伴。大家好，我是 Cobo Agentic Wallet 的产品负责人 Johnny。今天我想跟大家分享一件我们认为会重塑下一代加密经济的重要事情：为 AI Agent 的操作层打造一个可行、可信任的基础设施。它的目标是让未来越来越强大的大模型，能够在一个可控边界内替人类花费 Crypto。

今天我想跟大家讨论三个问题：

## 二、AI Agent 发展路径与链上支付需求

**Johnny**：我们先回顾一下过去几年 AI 的发展历程。

- **2023 年 — Chatbot**：AI 主要做的是回答问题，所有工作仍然需要人类执行。ChatGPT 进入主流大众视野。
- **2024 年 — Copilot**：AI 开始提出建议、制定行动计划，但每一步依旧需要人类逐步批准。Cursor、Claude Code 成为主流程序员的日常工作习惯。
- **2025 年 — Agent**：AI 能够留在后台，持续解决用户的复杂工作流。人类只需要定义工作范围和边界。标志性事件：Lovable 成为全民热点。
- **2026 年 — Autonomous Agent**：AI 不仅能执行工作流，还能持续自主探索和任务。但还有一个大痛点没解决：**如何让 AI 替人类花钱，完成代表人类意志的消费。**

两个核心命题：
1. 链上资金始终需要被管理，资金归属是否安全、是否具有确定性（链上资金不可逆）
2. 相关基础设施是否已经到位，能否辅佐 Agent 完成代币消费

**市场数据**：2025 年市场规模 72.9 亿美元，到 2034 年前约 40% 复合增长率。链上已有 AI Agent 约 250,000 个。

> Agent 操作链上资金已经不是未来时，而是现在时。

## 三、缺少可控边界时的两个案例

### 案例一：Send Override（静默覆盖）

当 Agent 执行 DeFi 操作时，虽然 Prompt 里明确告诉 Agent "只能花 100 美金"，但如果没有硬性约束，Agent 可能在执行过程中**悄悄修改交易金额**，而且不会主动告诉你。只有人类复盘时，Agent 才会承认"我做错了"。

> 自然语言能够表达常规意图，但**不具备强制约束权限**。如果让 AI 具备独立自主的资金转移权限，其实是相对可怕的。

### 案例二：Shadow Custody（影子托管）

因为采用 MPC 钱包架构，Agent 可能在 MPC 钱包外围再创建一个独立的 EOA 地址，先把钱转入这个 EOA，再发起合约交易。这样无论设置多少权限，都无法控制 MPC 以外的交易行为。

> 当缺失策略引擎、审计跟踪、一键冻结能力时，这些行为都是不可控的。

## 四、Agent 动用资金时的四类失控风险

1. **Prompt Injection**：Prompt 受到外界影响或模型幻觉，Agent 执行未经授权的交易
2. **Shadow Operations**：Agent 在人类看不见的地方创建子账户，执行潜在路径
3. **Unscoped Authority**：没有设置权限时，Agent 对资金有无限掌控能力。EOA 私钥泄露或幻觉可能导致一笔转出所有资金
4. **Zombie Permissions**：授权没有被撤销，合约后续发生系统性风险时，长期处在攻击面上

> 总结：当 Agent 开始动用资金的那一刻，信任就不再是一个应用层的事情，而必须提升到基础层面。

## 五、Cobo 的可信执行链路

```
submit（Agent 提交请求）
    ↓
approve（人类审批）
    ↓
自主签名（Agent 在 Pact 范围内执行）
    ↓
链上可追溯结果
```

三个独特方案：
1. **MPC 方案**：用于服务 AI Agent 场景
2. **Pact Authority**：可授权模型的执行层规范
3. **Recipe Skill Layer**：降低模型风险和幻觉

## 六、MPC：私钥安全与 2-of-2 签名模式

**核心问题**：私钥该由谁持有？

- 市面上大多数方案依赖 TEE 或临时 Session Token/API Key，存在单点风险
- Cobo 采用 2-of-2 threshold 模式：Cobo、Agent、Human 各自持有私钥分片，任意一方没有独立掌控资金转移的权限

**两个场景**：

1. **Agent + Cobo 共管**：人类批准 Pact 后，Agent 在 Pact 范围内自动操作，Cobo 根据 Pact 审批状态做安全把关和协同签名
2. **Human + Cobo 共管**：用户正常使用 Web3 钱包，大额资金操作由 Human 发起签名，Cobo 配合完成

> 没有单独一方可以随意转移资金；Agent 的所有执行一定受到 Pact 限制。

## 七、Pact：从意图到授权协议

**Pact = 告诉 Agent：你能做什么，不能做什么，在哪些环节必须停下来。**

四个核心要素：

1. **Intent（意图）**：期待 Agent 完成什么任务。例："以太坊低于 2000 美元时买入，高于 2500 美元时卖出。"
2. **Execution Plan（执行计划）**：AI 把意图转译成具体执行计划——调用哪个合约、执行多少 ETH、Token Pair 是什么。详细、透明、可供审计。
3. **Policy（风控约束）**：最核心的部分。包括预算、审批规则、白名单、链、Token、合约限制，精确到 ABI 参数级。每次执行都重复校验。
4. **Completion Condition（完成条件）**：Pact 不应该永久有效。超过 1000 美金后自动拦截，或在任务完成后自动 revoke。

---

## 八、Pact 的实际执行流程

1. 用户在智能体里表达执行意图
2. Agent 和 Cobo 接口通信，把意图转译成 Execution Plan，同步完成 Policy 和 Completion Condition 组装
3. 封装成 Pact，推送到移动端 App，用户审阅、修改、批准或拒绝
4. 审批后，Agent 在可控范围内完成交易

## 九、Recipe：让 Agent 知道如何把事情做对

- **Pact 定义边界，Recipe 赋予技能**
- Agent/大模型并不具备自主操作资金的能力，成功率不高
- Recipe = 预加载知识库 = "知识胶囊"
- 把合约地址、ABI 参数、安全风控边界条件全部打包
- 让 Agent 基于已验证的执行路径完成任务，而不是随意发挥

**已上线 Recipe**：Aave V3、Uniswap V3、Polymarket、Hyperliquid 等

## 十、多 Agent 架构与 Demo

- 一个 Human 可以有多个 Agent 共管资产，不同 Agent 资金彼此独立
- 可以创建多个钱包，delegate 给不同 Agent
- Trading Agent、DeFi Agent、各种场景 Agent，Policy 各不相同

**Demo 亮点**：Pact 审阅过程中加入 AI 决策辅助——高中低风险监控，帮助用户了解交易是否存在高风险行为。

## 十一、Q&A

### Q1：LLM 串在哪里？

**Richard**：你们的 LLM 是串在哪里的？客户要用你们家的服务，LLM 是放在你们这边吗？

**Johnny**：钱包服务于 Agent 场景。Agent 本身需要用户自己搭建——你需要有一个 Lovable 或 Claude Code 之类的 Agent 智能体，能执行复杂 Skill 安装等行为，才能使用这个产品。

### Q2：小额免密支付场景

**Wiki**：有没有考虑过小额免密支付？

**Johnny**：已经支持 x402 微支付协议，也计划支持 gasless 场景（用户只需要 USDC/USDT 就可以完成 API 服务支付）。

**Wiki 的想法**：针对单一支付类型（如 Token 用量不足时自动购买），金额不大、场景固定、风险低。一个人管理 3-4 个 Agent，余额不够时自动购买。

**Johnny 回应**：在 Pact 体系下可以实现——Policy 限制单次金额（如 < 100 美金）、限制转账地址（只允许 1-2 个 provider 地址）、限制频次（每天一笔）、设置总上限（如 1000 美金）。超过上限需要临时单笔授权。

### Q3：Agent 支付一定要在链上吗？

**Harold**：Agent 支付一定要发生在链上吗？大部分用户可能不在意去中心化，只是觉得方便就行。

**Johnny**：
- 主流支付场景一定还是偏向 Web2 的
- 但从 self-custody 和 Agent Economy 角度，链上是更自然的选择
- 链上是相对快速、高效的结算系统，绕不开 Web2 的信用卡 CVV、SWIFT 等
- 跨境支付和跨国结算已开始使用 USDC/USDT
- 未来 Web2 和 Crypto 各占一部分，Crypto 比重会持续变大

### Q4：Agent Wallet 和主钱包是否操作同一账户？

**Johnny**：是的，同一个 wallet、同一个地址。两个控制人：Agent 和 Human。区别在于 Agent 需要先经过 Pact 人类审批，只能操作 Pact 范围内的资金。Human 拥有绝对掌控权。

如果需要额外保护，可以分两个钱包：一个常规 EOA 放更多资金，另一个 Agentic Wallet 放小额资金给 Agent 操作。

### Q5：Human-in-the-loop 与未来 Agent 自主执行

**Johnny**：
- 短期（1-2 年）：Human-in-the-loop 不可或缺，Agent 基础设施还不健全
- 长期（5-10 年）：当 Agent 足够智能、生态搭建起来后，可能出现 Agent 自发支付、Agent 雇佣 Agent 的场景

### Q6：Agent 掉线或服务不可用时的兜底方案

**Johnny**：MPC 钱包做了冗余设计。Agent 挂掉后，Agent 的私钥分片会 re-share 给移动端设备。只要移动端设备的 share 分片经过本地存储或云端备份，即使 Agent 私钥丢失，资金也不会损失。移动端设备也丢了的话，可以通过 Google 邮箱或 Apple ID 从云端恢复。

> 只要 follow 所有安全设计规范，bottom line 是钱肯定不会丢。但 Agent 能力可能需要重新配置。

### Q7：意图执行过程中的攻击防范

**Johnny**：两个层面：
1. **Recipe Layer**：把主流链上行为包装成 Recipe，减少大模型幻觉
2. **Pact + 策略校验**：对可调用合约、Token Pair、交易金额、滑点、minimum received 等关键参数进行限制，每一步都和授权边界校验
